# Tessera 제품 가치 문서

> **Tessera**(테세라)는 작업 코드네임입니다. tesserae(모자이크를 이루는 작은 타일)에서 따왔으며,
> window/pane/tab으로 화면을 구성하는 멀티플렉서의 성격을 반영합니다. 원하는 제품명으로 자유롭게 변경할 수 있습니다.

> 한 줄 정의: **macOS용 Electron 기반 윈도우 멀티플렉서** — 터미널·브라우저·텍스트 편집기·Claude Code GUI를
> window/pane/tab으로 묶고, workspace 단위로 호스트 또는 컨테이너 환경에서 실행한다.

## 제품 가치

### V1: 통합된 단일 작업 표면 (Unified Single Work Surface)

- **유형**: 구체적
- **설명**: 개발에 필요한 4종 컴포넌트(터미널, 인터넷 브라우저, 텍스트 편집기, Claude Code GUI)를
  window/pane/tab 레이아웃 하나로 묶는다. 개발자는 별도 앱 사이를 오가지 않고, 하나의 멀티플렉서 표면 안에서
  모든 작업을 배치·전환한다. 컨텍스트 전환 비용을 줄이는 것이 핵심이다.
  여러 작업 단위(workspace)를 두고 쓸 때도, 단일 앱 창의 workspace 목록에서 전환해 같은 표면 안에서 작업 단위 사이를 오간다.
- **측정 지표**: 4종 컴포넌트 전부를 단일 window/pane/tab 레이아웃 안에서 배치·전환·재정렬할 수 있는가.
- **근거 요구사항**: #1(4종 컴포넌트 지원), #5(window/pane/tab 레이아웃), #12(포커스된 pane 전체화면(zoom) 토글, 영속), #13(워크스페이스 목록 표시 및 전환)

### V2: 환경 선택의 자유 (Freedom of Execution Environment)

- **유형**: 추상적
- **설명**: 같은 도구와 동일한 조작 방식을 유지한 채, 작업 단위(workspace)마다 실행 환경을
  호스트 머신과 컨테이너 중에서 선택할 수 있다. 컨테이너 백엔드도 호스트만큼 native하게 동작하여,
  격리된 환경이 "원격에 붙은 부속물"처럼 느껴지지 않는다. 개발자는 작업 성격에 맞는 환경을
  자유롭게 고르되 학습 비용을 새로 치르지 않는다.
  워크스페이스의 기본 환경 선택에 더해, 컨테이너 워크스페이스 안에서는 선택적으로 **host 전용 영역**을
  두어 일부 도구만 호스트 backend에서 실행할 수 있다 — 컨테이너 격리를 기본으로 유지하면서,
  필요한 도구에 한해 호스트로 빠져나가는 명시적이고 경계가 분명한 탈출구다.
- **방향성**: 환경(격리 수준)은 선택의 문제이지 도구 교체의 문제가 아니어야 한다.
  격리에서 빠져나가는 것 또한 선택이되, 항상 명시적이고 경계가 눈에 보여야 한다.
- **근거 요구사항**: #2(호스트/컨테이너 백엔드 native 지원), #6(workspace 단위 백엔드 선택), #7(pane/tab은 자신이 속한 영역의 backend 환경에서 실행), #11(컨테이너 workspace의 선택적 host 전용 영역)

### V3: 격리를 깨지 않는 인증 경험 (Seamless Auth Across the Isolation Boundary)

- **유형**: 구체적
- **설명**: 컨테이너 내부 작업이 OAuth 등으로 브라우저를 필요로 할 때, 호스트에서 실행되는
  Tessera 브라우저를 통해 인증이 끝까지 완결된다. 컨테이너 격리가 실제 인증 흐름(외부 IdP 로그인,
  localhost 콜백 수신)을 가로막지 않는다. 개발자는 호스트에 이미 로그인된 세션과 신원을 그대로 재사용한다.
- **측정 지표**: 컨테이너 안에서 시작한 OAuth 흐름이 호스트 브라우저를 거쳐 토큰 수신까지 끊김 없이 완결되는가.
- **근거 요구사항**: #3(브라우저는 항상 호스트에서 실행), #4(컨테이너의 브라우저 open 요청을 구현된 브라우저로 라우팅)

### V4: 작업 손실 없는 복원력 (Crash Resilience Without Work Loss)

- **유형**: 구체적
- **설명**: 백엔드(호스트 프로세스/컨테이너)나 애플리케이션이 비정상 종료되어도, 개발자가 잃는 것이 없어야 한다.
  편집기와 Claude Code의 마지막 상태, 터미널의 마지막 화면(명령 실행 및 스크롤), 브라우저의 탭별 URL이
  복원된다. 크래시는 사고이지 작업·맥락의 소멸 사유가 아니다.
- **측정 지표**: 백엔드/앱 비정상 종료 후, 각 컴포넌트의 마지막 상태가 복원되는가(데이터 손실 0 목표).
- **근거 요구사항**: #8(백엔드 종료 시 편집기·Claude Code 상태 복원), #9(백엔드 종료 시 터미널 화면 복원), #10(앱 종료 시 브라우저 탭별 URL 복원)
