# 테스트 문서: 크로스-격리 브라우저 라우팅

- **Test ID**: T-3
- **대상 PRD**: PRD-3 (`tessera-prd-browser-routing.md`)

## 검증 대상 AC

- AC3.1: 브라우저는 항상 호스트에서 실행
- AC3.2: 컨테이너의 브라우저 open 요청을 호스트 브라우저로 전달 (방향 A)
- AC3.3: OAuth localhost 콜백 포트의 호스트→컨테이너 포워딩 (방향 B)
- AC3.4: 인증 루프 완결 및 토큰 수신
- AC3.5: 라우팅 채널의 workspace/컨테이너별 격리

## 테스트 시나리오

### 시나리오 1: 컨테이너 workspace에서도 브라우저는 호스트 실행

- **사전 조건**: container workspace 기동
- **실행 단계**: 브라우저 tab을 열고 페이지 탐색, 렌더링 프로세스 위치 확인
- **기대 결과**: 브라우저가 호스트 Chromium에서 동작, 컨테이너 내부에 브라우저 미기동
- **검증 AC**: AC3.1

### 시나리오 2: 방향 A — URL 전달

- **사전 조건**: container workspace, 터미널 사용 가능
- **실행 단계**: 컨테이너 터미널에서 `xdg-open https://example.com`(및 `$BROWSER` 기반 호출) 실행
- **기대 결과**: 호스트 Tessera 브라우저의 새 tab에 해당 URL이 열림
- **검증 AC**: AC3.2

### 시나리오 3: 방향 B — 콜백 포워딩

- **사전 조건**: 컨테이너 내부에서 `localhost:CB` 콜백 리스너를 띄우는 OAuth 모의 흐름
- **실행 단계**: 호스트 브라우저가 `http://localhost:CB/callback?...`로 리다이렉트되도록 유도
- **기대 결과**: 리다이렉트 요청이 호스트→컨테이너 포워딩을 통해 컨테이너 리스너에 도달
- **검증 AC**: AC3.3

### 시나리오 4: end-to-end 인증 완결

- **사전 조건**: 실제 또는 모의 IdP, container workspace
- **실행 단계**: 컨테이너 내부 앱에서 OAuth 시작 → 호스트 브라우저 로그인 → 콜백 수신
- **기대 결과**: 컨테이너 프로세스가 유효 토큰을 수신하고 인증이 완료됨
- **검증 AC**: AC3.4 (AC3.2·AC3.3 결합 검증)

### 시나리오 5: 동시 인증 격리

- **사전 조건**: container workspace 2개 동시 기동, 각기 OAuth 흐름 진행
- **실행 단계**: 두 컨테이너에서 동시에 인증 시작, 각 콜백 발생
- **기대 결과**: 각 콜백이 자신의 컨테이너로만 전달, 포트 충돌·오배달 없음
- **검증 AC**: AC3.5
