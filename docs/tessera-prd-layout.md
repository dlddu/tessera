# PRD: 컴포넌트 & 레이아웃 시스템 (Component & Layout System)

- **PRD ID**: PRD-1
- **참조 가치 문서**: `tessera-values.md`

## 달성 가치

- **V1: 통합된 단일 작업 표면** — 4종 컴포넌트를 window/pane/tab 하나의 레이아웃 체계로 묶어,
  도구 간 컨텍스트 전환 없이 작업하게 한다.

## 개요

Tessera의 화면 모델은 3계층이다.

- **Window**: 한 workspace의 최상위 레이아웃 컨테이너(pane/tab을 담는다). workspace와 1:1로 대응한다(백엔드 소속은 PRD-2에서 정의). **앱은 단일 창**으로 동작하며, 그 창 안에서 workspace 목록과 함께 **활성 workspace의 window 하나**를 표시한다 — 여러 workspace의 window를 동시에 나란히 표시하지 않고, 목록에서 전환한다(AC1.7).
- **Pane**: window를 수직/수평으로 분할한 타일 영역. 각 pane은 독립적으로 크기 조절 가능.
- **Tab**: pane 내부에서 전환되는 단위. 하나의 tab은 정확히 하나의 컴포넌트 surface를 담는다.

컴포넌트 surface는 4종이다.

| 컴포넌트 | 구현 방향 | 비고 |
|---|---|---|
| 터미널 | xterm.js + PTY (호스트 node-pty / 컨테이너 exec) | 스크롤백 보유 |
| 인터넷 브라우저 | Electron WebContentsView(실 Chromium) | **항상 호스트 실행**(PRD-3) |
| 텍스트 편집기 | 인앱 코드 에디터(CodeMirror 6) + 파일 버퍼 | 버퍼/커서 상태 보유 |
| Claude Code GUI | Claude Code 세션을 감싸는 GUI surface | 세션 상태 보유 |

## Acceptance Criteria

### AC1.1: 4종 컴포넌트를 pane/tab에 배치

- **설명**: 사용자는 새 tab을 생성할 때 4종 컴포넌트(터미널, 브라우저, 텍스트 편집기, Claude Code GUI) 중
  하나를 선택해 해당 surface를 띄울 수 있다. 한 pane은 서로 다른 컴포넌트 타입의 tab을 섞어 가질 수 있다.
- **달성 가치**: V1
- **검증 방법**: 빈 window에서 각 컴포넌트 타입의 tab을 1개씩 생성하고, 4종 surface가 모두 정상 렌더·동작하는지 확인.

### AC1.2: window 분할로 pane 생성 및 비율 조절

- **설명**: 사용자는 window를 수직/수평으로 분할해 pane을 만들 수 있고, pane 경계를 드래그하거나
  단축키로 비율을 조절할 수 있다. 중첩 분할(분할된 pane을 다시 분할)을 지원한다.
- **달성 가치**: V1
- **검증 방법**: 단일 pane을 2×2 그리드로 중첩 분할한 뒤, 각 pane 비율을 조절하고 레이아웃이 깨지지 않는지 확인.

### AC1.3: tab 전환·재정렬·pane 간 이동

- **설명**: pane 내에서 tab을 전환·재정렬할 수 있고, tab을 다른 pane으로 드래그해 이동할 수 있다.
  pane의 마지막 tab을 닫으면 해당 pane이 정리되고 레이아웃이 재배치된다.
- **달성 가치**: V1
- **검증 방법**: tab을 한 pane에서 다른 pane으로 이동, 순서 변경, 마지막 tab 종료 시 pane 정리까지 순차 확인.

### AC1.4: 키보드 중심 레이아웃 조작

- **설명**: pane 분할/이동/포커스 전환, tab 생성/전환/종료를 마우스 없이 키보드 단축키로 수행할 수 있다.
  단축키 체계는 일관되며 사용자가 확인할 수 있도록 노출된다.
- **달성 가치**: V1
- **검증 방법**: 마우스를 사용하지 않고 단축키만으로 2×2 레이아웃 구성 → 포커스 이동 → tab 전환 시나리오 완수.

### AC1.5: 레이아웃 직렬화 및 재구성

- **설명**: 현재 window/pane/tab 구조(각 tab의 컴포넌트 타입과 배치)를 직렬화해 저장하고, 그대로
  재구성할 수 있다. 이는 V4의 **컴포넌트 콘텐츠 상태 복원**(편집기 버퍼·터미널 화면·브라우저 URL)의 기반이 된다.
- **달성 가치**: V1
  - 콘텐츠 단위 복원은 PRD-4가 담당. 본 AC는 "레이아웃 골격"의 저장/재구성까지를 범위로 한다.
- **검증 방법**: 복잡한 레이아웃을 저장 → 앱 재시작(정상 종료) → 동일한 pane/tab 골격으로 재구성되는지 확인.

### AC1.6: 포커스된 pane 전체화면(zoom) 토글

- **설명**: 사용자는 포커스된 pane을 단축키(⌃⌘⏎) 한 번으로 window 전체 크기로 확대(zoom)하고, 같은
  단축키나 Esc로 원래 레이아웃으로 즉시 복귀할 수 있다. 확대는 분할 구조·pane 비율·tab 구성 등 기존
  레이아웃을 바꾸지 않고 그대로 보존했다가 복원하되, **확대 여부 자체는 레이아웃 골격의 일부로 저장·복원**되어
  앱을 재시작해도 같은 zoom 상태로 되돌아온다(AC1.5의 직렬화에 `zoomedPaneId`로 포함). 확대는 **포커스를
  따라간다**: 확대 중 포커스를 이웃 pane으로 옮기면 그 pane이 대신 확대된다. 확대 중에는 해당 pane이 활성
  상태임이 시각적으로 드러나고(zoom 배지), 나머지 pane은 숨겨진다(단 마운트는 유지되어 터미널 PTY·버퍼가
  보존된다). tab 전환은 확대 상태에서도 가능하다.
- **달성 가치**: V1
  - 한 작업에 집중할 때 다른 타일에 화면을 빼앗기지 않게 하여, 단일 작업 표면 안에서 컨텍스트 전환 비용을 더 낮춘다.
  - AC1.2의 pane 비율 조절(분할 구조 자체의 영구적 크기 변경)과 달리, 본 AC는 분할 구조를 보존한 채 표시만 한
    pane으로 좁히는 **가역적** 확대다. 단, 그 가역적 상태도 골격의 일부로 영속한다.
- **검증 방법**: 2×2 레이아웃에서 한 pane을 확대 → window 전체를 채우는지 확인 → 확대 상태에서 tab 전환 동작 확인
  → 포커스를 이웃 pane으로 이동 시 확대 대상이 따라 바뀌는지 확인 → 단축키/Esc로 복귀 시 이전 2×2 분할·비율·tab
  구성이 그대로 복원되는지 확인 → **확대한 채로 앱을 재시작했을 때 동일한 zoom 상태로 복원되는지 확인**.

### AC1.7: 워크스페이스 목록 표시 및 전환

- **설명**: 앱은 단일 창에서 사용자가 보유한 workspace들의 **목록**을 항상 확인·접근할 수 있게 제공하고,
  목록에서 하나를 선택하면 활성 workspace가 즉시 전환되어 해당 workspace의 window(pane/tab 레이아웃)가 표시된다.
  전환은 마우스와 키보드 단축키 양쪽으로 가능하다. 전환은 **표시 대상만 바꾸는 것**이며, 비활성 workspace의
  레이아웃·상태는 보존된다(다시 전환하면 그대로 복귀). 여러 workspace의 window를 동시에 나란히 표시하는 것은
  본 AC의 범위가 아니다.
- **달성 가치**: V1
  - 별도 앱/창 사이를 오가지 않고 단일 표면 안에서 작업 단위(workspace) 간 이동까지 끝내게 하여, 컨텍스트 전환 비용을 더 낮춘다.
  - AC1.3의 pane/tab 전환(한 workspace **안에서**의 이동)보다 상위 계층인 **workspace 사이**의 전환을 정의한다.
- **검증 방법**: 2개 이상의 workspace를 생성한 뒤, 목록에서 다른 workspace를 선택 → 활성 window가 즉시 그 workspace의
  레이아웃으로 바뀌는지 확인 → 이전 workspace로 되돌아갔을 때 pane/tab 구성·상태가 보존되는지 확인 → 마우스와 단축키
  양쪽으로 전환되는지 확인.

## 의존 관계

- PRD-2(백엔드 & 워크스페이스): pane/tab이 실행되는 backend/env를 제공.
- PRD-3(브라우저 라우팅): 브라우저 컴포넌트의 실행 위치(항상 호스트) 제약을 정의.
- PRD-4(상태 복원): AC1.5의 레이아웃 골격 위에 컴포넌트 콘텐츠 상태 복원을 얹음.
