# PRD: 크로스-격리 브라우저 라우팅 (Cross-Isolation Browser Routing)

- **PRD ID**: PRD-3
- **참조 가치 문서**: `tessera-values.md`

## 달성 가치

- **V3: 격리를 깨지 않는 인증 경험** — 컨테이너 내부의 브라우저 open(OAuth 등) 요청을 호스트의 Tessera
  브라우저로 라우팅하여, 격리를 유지한 채 인증을 완결한다.

## 개요

브라우저 컴포넌트는 backend와 무관하게 **항상 호스트에서 실행**된다. 문제는 컨테이너 내부 프로세스가
인증 등을 위해 브라우저를 열려고 할 때다. 두 방향의 다리가 필요하다.

```
[컨테이너 내부]                         [호스트]
  앱이 OAuth 시작                         Tessera 브라우저(실 Chromium)
  → auth URL open 시도                     ↑
        │ (방향 A) URL 전달                 │
        └──────────────────────────────────┘
                                            │ 사용자 IdP 로그인
                                            ▼
  콜백 리스너(localhost:CB) ◀──────── localhost:CB 리다이렉트
        ▲ (방향 B) 콜백 포트 포워딩         (docker proxy 등)
        └───────────────────────────────────┘
  → 토큰 수신, 인증 완료
```

- **방향 A (URL 전달, 컨테이너 → 호스트)**: 컨테이너 안에서 `xdg-open`/`open`/`$BROWSER` 호출이나
  "아래 URL을 브라우저에서 여세요" 출력이 발생하면, 그 URL을 호스트의 Tessera 브라우저로 전달해 연다.
  구현은 컨테이너에 주입한 `xdg-open`/`open` shim과 `BROWSER` 환경변수, 그리고 호스트 앱이 노출하는
  라우팅 채널(소켓/RPC)을 통한다.
- **방향 B (콜백 포워딩, 호스트 → 컨테이너)**: OAuth는 보통 `http://localhost:CB/callback`으로 리다이렉트한다.
  그런데 콜백 리스너는 컨테이너 내부에 있고, 호스트 브라우저의 localhost는 호스트의 것이다. 따라서 해당
  콜백 포트를 호스트 → 컨테이너로 포워딩(docker proxy/port publish 등)해, 호스트 브라우저의 리다이렉트가
  컨테이너 리스너에 도달하게 한다. 이로써 인증 루프가 닫힌다.

> host backend workspace에서는 브라우저가 이미 호스트에 있고 localhost가 공유되므로 방향 A/B 모두 자명하게 동작한다.
> 본 PRD의 라우팅 메커니즘은 주로 container backend workspace를 위한 것이다.

## Acceptance Criteria

### AC3.1: 브라우저는 항상 호스트에서 실행

- **설명**: workspace의 backend가 container여도, 브라우저 컴포넌트 surface는 호스트의 실 Chromium에서 실행된다.
  컨테이너 내부에 별도 브라우저를 띄우지 않는다.
- **달성 가치**: V3
- **검증 방법**: container workspace에서 브라우저 tab을 열고, 렌더링 프로세스가 호스트에서 동작함을 확인.

### AC3.2: 컨테이너의 브라우저 open 요청을 호스트 브라우저로 전달 (방향 A)

- **설명**: 컨테이너 내부에서 `xdg-open <url>`/`open <url>` 호출 또는 `$BROWSER` 기반 실행이 발생하면,
  해당 URL이 호스트 Tessera 브라우저의 새 tab으로 열린다. URL 출력만 하는 도구의 경우에도 동일 경로로
  열 수 있는 수단(예: shim 또는 사용자 액션)을 제공한다.
- **달성 가치**: V3
- **검증 방법**: 컨테이너에서 `xdg-open https://example.com` 실행 시 호스트 브라우저에 해당 페이지가 열리는지 확인.

### AC3.3: OAuth localhost 콜백 포트의 호스트→컨테이너 포워딩 (방향 B)

- **설명**: 컨테이너 내부 OAuth 흐름이 `localhost:CB` 콜백 리스너를 띄우면, Tessera가 해당 포트를
  호스트 → 컨테이너로 포워딩한다. 호스트 브라우저가 `http://localhost:CB/...`로 리다이렉트될 때
  요청이 컨테이너 리스너에 도달한다.
- **달성 가치**: V3
- **검증 방법**: 컨테이너에서 localhost 콜백을 쓰는 OAuth 모의 흐름을 실행하고, 호스트 브라우저의 콜백 리다이렉트가 컨테이너 리스너에 도달하는지 확인.

### AC3.4: 인증 루프 완결 및 토큰 수신

- **설명**: 방향 A·B가 결합되어, 컨테이너에서 시작한 OAuth가 호스트 브라우저 로그인을 거쳐 컨테이너
  프로세스의 토큰 수신까지 끊김 없이 완료된다.
- **달성 가치**: V3
- **검증 방법**: 실제(또는 모의) IdP 대상으로 컨테이너 내 인증을 end-to-end 수행하고, 컨테이너 프로세스가 유효 토큰을 획득하는지 확인.

### AC3.5: 라우팅 채널의 workspace/컨테이너별 격리

- **설명**: 여러 컨테이너 workspace가 동시에 동작해도, 각 컨테이너의 브라우저 open 요청과 콜백 포워딩이
  서로 섞이지 않고 올바른 컨테이너로 라우팅된다(콜백 포트 충돌·오배달 방지).
- **달성 가치**: V3
- **검증 방법**: 두 개의 container workspace에서 동시에 인증 흐름을 진행하고, 각 콜백이 자신의 컨테이너로만 전달되는지 확인.

## 의존 관계

- PRD-2: 라우팅 대상이 되는 container backend와 그 생명주기를 제공.
- PRD-1: 라우팅된 URL이 열리는 브라우저 컴포넌트 surface를 제공.
- **보안 주의**: 라우팅 채널과 포트 포워딩은 임의의 외부가 아니라 해당 workspace 컨테이너로만 제한되어야 한다(AC3.5).
