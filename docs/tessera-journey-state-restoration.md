# 사용자 여정 J4 — 크래시에서 작업 복원

- **여정 ID**: J4
- **참조 가치 문서**: `tessera-values.md`
- **여정 인덱스**: `tessera-user-journeys.md`
- **관련 PRD / 테스트**: PRD-4 (`tessera-prd-state-restoration.md`) / T-4 (`tessera-test-state-restoration.md`)

## 페르소나 · 상황

- **페르소나**: macOS에서 일하는 개발자 (주 페르소나 정의는 `tessera-user-journeys.md` 참조).
- **상황**: 장시간 작업 중 백엔드(컨테이너/호스트 백엔드 프로세스)나 Tessera 앱 전체가 비정상 종료된다.
  개발자는 미저장 변경과 맥락을 잃을까 걱정한다.

## 달성 가치

- **V4(작업 손실 없는 복원력)** — 주
- **V1(통합된 단일 작업 표면)** — 부 (AC1.5 레이아웃 골격 기반)

## 트리거

백엔드 또는 앱의 비정상 종료.

## 사전 상태

편집기 미저장 버퍼, 진행 중인 Claude Code 세션, 명령이 쌓인 터미널 스크롤백, 여러 개의 브라우저 탭.

## 복원 원칙 — "사용 가능한 상태"여야 한다

복원의 목표는 **실제 사용 가능한 상태**다. 백엔드(컨테이너/호스트 백엔드 프로세스)에 의존하는
컴포넌트(터미널·편집기·Claude Code)는, **죽은 백엔드가 다시 살아난 뒤 그 위에 상태를 재적용(rehydrate)해야
비로소 복원이 완료**된다. 백엔드가 죽어 있는 동안 유지되는 마지막 화면은 맥락·미저장 변경을 잃지 않기 위한
**중간 보존 뷰**일 뿐, 그 자체로 동작하는 상태가 아니다.

> 브라우저는 backend와 무관하게 항상 호스트에서 실행되므로(PRD-3) 이 제약을 받지 않는다.
> 따라서 브라우저 탭 URL 복원(AC4.4)은 백엔드 종료(시나리오 A)가 아니라 앱 종료(시나리오 B)에서 다룬다.

## 단계

| 단계 | 사용자 행동 · 시스템 반응 | 근거 AC | 시각화 |
|------|---------------------------|---------|--------|
| 1 | **(시나리오 A · 백엔드 종료 + 중간 보존)** 컨테이너/호스트 백엔드 프로세스가 비정상 종료. 앱은 생존하며 단절을 감지하고, 보유한 마지막 상태로 화면을 보존(맥락·미저장 변경 유지). 단 이는 **읽기 전용 중간 보존 뷰**이며 아직 사용 가능한 복원이 아니다. | AC4.1, AC4.2, AC4.3 | ✅ [M-J4-S1](./mockups/M-J4-S1.html) |
| 2 | **(A · 백엔드 재기동)** 죽은 컨테이너/호스트 백엔드를 다시 살린다(자동 또는 사용자 트리거). 사용 가능한 복원은 백엔드가 살아난 **이후에만** 성립한다. | AC4.5 | ✅ [M-J4-S2](./mockups/M-J4-S2.html) |
| 3 | **(A · 상태 재적용 → 사용 가능)** 살아난 백엔드 위로 상태를 **재적용(rehydrate)**해 실제 사용 가능 상태로 복원: 편집기 버퍼·커서 재적용, Claude Code 세션 재개, 터미널은 **새 PTY**를 띄우고 보존된 화면·스크롤백을 그 위 히스토리로 표시(동작하는 셸 복귀). | AC4.1, AC4.2, AC4.3 | ✅ [M-J4-S3](./mockups/M-J4-S3.html) |
| 4 | **(시나리오 B · 앱 종료)** Tessera 앱 전체가 비정상 종료 → 재시작. 앱 메모리가 사라졌으므로 **호스트 영속 저장소**에서 레이아웃 골격 + 컴포넌트 상태를 로드하고, 브라우저 탭별 URL·순서를 복원(브라우저는 호스트 실행이라 backend와 독립). 이어서 백엔드를 재기동하고 2~3과 동일하게 재적용한다. | AC1.5, AC4.4, AC4.5 | ✅ [M-J4-S4](./mockups/M-J4-S4.html) |
| 5 | **(공통)** 영속 저장은 backend/app 수명과 **독립** — 컨테이너를 `rm` 하거나 앱이 죽어도 상태가 호스트에 남아, 새로 띄운(심지어 새) 백엔드 위로 복원 가능. | AC4.5 | ✅ [M-J4-S5](./mockups/M-J4-S5.html) |
| 6 | **(공통)** 재적용 시 보존 상태와 실제(디스크/새 백엔드)의 **불일치를 감지·고지**(예: 디스크 파일이 그새 변경), 사용자가 동기화/선택해 손실 없이 정리. | AC4.6 | ✅ [M-J4-S6](./mockups/M-J4-S6.html) |

> **의존**: 백엔드 재기동 메커니즘 자체는 PRD-2의 backend 생명주기 관리(AC2.6)에 기댄다.
> 본 여정은 그 위에서 "상태 재적용을 통한 사용 가능 복원"을 다룬다.

## 완료 상태

백엔드/앱 비정상 종료 후, 죽은 백엔드를 다시 살리고 그 위로 상태를 재적용해 4종 컴포넌트가
**실제 사용 가능한 상태**로 복원된다(데이터 손실 0 목표) → **V4 달성**.
복원 결과는 동결된 화면이 아니라 곧바로 이어서 작업할 수 있는 살아있는 상태이며,
크래시가 작업·맥락의 소멸 사유가 되지 않는다.
